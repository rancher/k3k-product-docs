如何选择共享模式和虚拟模式

本指南帮助您为虚拟集群选择合适的模式：共享 或 虚拟。

如果您不确定，请从 共享模式 开始 - 这是默认设置，适合大多数常见场景。

共享模式（默认）

最佳适用对象：

  • 希望快速运行工作负载而无需管理Kubernetes内部的开发人员

  • 需要对所有工作负载具备可见性与控制能力的平台团队

  • 需要访问主机级资源的用户（例如，GPU）

在 共享模式 下，虚拟集群运行自己的K3s服务器，但依赖主机来执行工作负载。虚拟 kubelet 同步资源，实现轻量级、快速部署，并支持集群资源隔离。有关架构的更多详细信息。

按角色划分的用例

开发人员

"我正在构建一个应该在虚拟集群外部暴露的Web应用程序。"
→ 使用 共享模式。它允许您 暴露 您的应用程序。

数据科学家：

"`I need to run Jupyter notebooks that leverage the cluster’s GPU.`"
→ 使用 共享模式。它在保持低开销的同时提供对物理设备的访问。

平台管理员

"我想从一个中心位置监控和保护所有租户的工作负载。"
→ 使用 共享模式。主机级代理（例如，可观测性、策略执行）在所有虚拟集群中工作。

安全工程师

"我需要在所有工作负载中执行安全策略，例如网络策略或运行时扫描。"
→ 使用 共享模式。该平台可以在不绕过租户的情况下全局执行策略。

"我需要测试一个新的准入控制器或策略引擎。"
→ 如果它的范围限于您的虚拟集群，请使用 共享模式。您可以运行像 Kubewarden 这样的工具，而不会影响主机。

CI/CD 工程师

"我想在每次管道运行时快速启动一次性虚拟集群，并且资源成本低。"
→ 使用 共享模式。它部署迅速，非常适合短期且受名称空间约束的环境。

虚拟模式

最佳适用对象：

  • 需要完全 Kubernetes 隔离的高级用户

  • 测试实验性或集群范围功能的开发人员

  • 需要控制整个 Kubernetes 控制平面的用例

在 虚拟模式 中，虚拟集群运行其自己的隔离 Kubernetes 控制平面。它支持不同的 CNI 和 API 配置 --，非常适合深入实验或高级工作负载。有关 架构 的更多详细信息。

按角色划分的用例

开发人员

"我需要测试一个在主机集群中禁用的新 Kubernetes 功能开关。"
→ 使用 虚拟模式。您可以配置自己的控制平面标志和 API 功能。

平台管理员

"我们正在测试不同 Kubernetes 版本的升级，包括新的 API 行为。"→ 使用虚拟模式。您可以运行不同的 Kubernetes 版本，并安全地验证升级路径。

网络工程师

"我正在评估一个新的 CNI，它需要对集群的网络进行全面控制。"
→ 使用 虚拟模式。您可以运行一个独立的 CNI 堆栈，而不会影响主机或其他租户。

安全工程师

"我正在测试一个新的准入控制器和策略引擎，然后再在整个集群中推广。"
→ 如果您需要测试集群范围的策略、自定义准入流程或具有全面控制的高级扩展，请使用 虚拟模式。

附加信息

如果您正在评估更高级的用例或想要更深入的比较，请查看 架构文档 中的完整权衡分析。
